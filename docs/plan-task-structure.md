# Plan Task Structure

本文记录面向研发和后续 `code` 阶段的 `plan.md` 建议结构。

如果本文与代码不一致，以代码为准。

## `plan.md` 的定位

当已经有了 `refine` 和 `design` 之后，`plan.md` 不应该重复回答：

- 需求是什么
- 为什么这么设计

`plan.md` 应该只回答一件事：

- 接下来准备怎么执行

因此三份文档的职责建议明确分工为：

- `refine`：收敛需求，回答“做什么”
- `design`：收敛设计，回答“为什么这么做、系统怎么配合”
- `plan.md`：收敛执行，回答“先做什么、后做什么、每一步做到什么算完成”

## 目标章节

面向研发和后续 `code` 阶段，`plan.md` 当前建议采用 5 个章节：

1. `实施策略`
2. `任务拆分`
3. `执行顺序`
4. `验证计划`
5. `阻塞项与风险`

其中核心是：

- `任务拆分`
- `执行顺序`

## 每章职责

### `实施策略`

这一章只做执行总述，不讲设计原理。

建议回答：

- 本次按什么维度推进
- 是按系统推进、按链路推进，还是按能力点推进
- 先收哪一层，后收哪一层

### `任务拆分`

这是 `plan.md` 最核心的一章。

这里必须把方案拆成稳定的任务单元，例如：

- `T1`
- `T2`
- `T3`

每个任务既要让研发读得懂，也要让后续 `code` 阶段可以稳定消费。

### `执行顺序`

这一章专门说明：

- 谁先做
- 谁后做
- 哪些可以并行
- 哪些必须等上游完成
- 哪些属于联调收口任务

注意这章和 `design` 里的“系统依赖关系”不同：

- `design` 讲为什么依赖
- `plan` 讲怎么排执行顺序

### `验证计划`

因为项目规模大、复杂度高，这一章建议保持最小验证原则，不扩展成重测试清单。

当前建议：

- 默认按受影响 package 做编译验证
- 不要求在 `plan.md` 中展开完整测试矩阵
- 把跨系统联调、灰度、回归建议留给 `design` 中的 QA 输入或后续专项文档

也就是说，这里的验证计划更偏“研发自证最小闭环”，而不是“完整测试方案”。

### `阻塞项与风险`

这一章记录：

- 当前有哪些前置条件未满足
- 哪些任务依赖外部资源或跨团队协作
- 哪些不确定性会影响实施顺序或实现方式

它的作用是防止 `plan.md` 看起来可执行，但真正落地时仍然大量依赖临场判断。

## 任务单元的推荐结构

每个任务建议固定使用下面这些字段：

- `任务ID`
- `任务名称`
- `目标系统/仓库`
- `目标`
- `依赖`
- `修改范围`
- `实施动作`
- `完成定义`
- `验证方式`

如果后续需要更压缩的 schema，最小也建议保留：

- `目标系统/仓库`
- `依赖`
- `修改范围`
- `实施动作`
- `完成定义`

因为从 `code` 视角看，这几项是最关键的执行输入。

## 验证方式的默认约束

针对当前仓库和类似的大型项目，`plan.md` 中的验证方式建议收敛为最小粒度：

- 默认写“按受影响 package 编译通过”
- 如果是多仓库任务，则按各自受影响 package 分别编译
- 不默认要求补充完整测试用例列表
- 不默认要求全量测试、全量回归或大而全的联调清单

这样做的原因是：

- 项目大而复杂，重验证内容写进 `plan.md` 反而会稀释执行主线
- 对自动执行阶段来说，最稳定、最通用的验证输入就是受影响 package 的编译闭环
- 更重的测试建议应留在 `design` 的 QA 输入，或者留给专项测试计划

## 推荐示例

```md
## 任务拆分

### T1 后端主链路改造
- 目标系统/仓库：server
- 目标：补齐主链路上的新字段处理与状态流转
- 依赖：无
- 修改范围：
  - internal/service/xxx
  - internal/repo/yyy
- 实施动作：
  - 扩展请求处理逻辑
  - 补齐状态更新与兜底
- 完成定义：
  - 主链路改动完成并可继续对接下游任务
- 验证方式：
  - 受影响 package 编译通过

### T2 RPC 协议对齐
- 目标系统/仓库：rpc-idl
- 目标：补齐上下游协议字段
- 依赖：T1 设计确认
- 修改范围：
  - idl/xxx.thrift
- 实施动作：
  - 增加字段
  - 保持兼容
- 完成定义：
  - 协议字段语义与上下游对齐
- 验证方式：
  - 受影响 package 编译通过
```

## 一句话结论

`design.md` 负责讲清方案，`plan.md` 负责把方案拆成可执行任务。

对当前这类大项目，`plan.md` 的验证部分应保持克制，默认以“受影响 package 编译通过”作为最小验证闭环。
