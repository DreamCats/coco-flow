# Plan

## 目标

- 为多规格竞拍商品标题补充规格名前缀
- 保证不同返回路径标题规则一致

## 前置依赖

- 无跨仓依赖
- 依赖现有规格信息可用，无需新增数据源

## 执行切片

### Slice 1

- 改动范围：主标题组装路径
- 主要文件/模块：普通竞拍标题生成逻辑
- 预期产出：多规格商品展示 `#规格名 商品标题`
- 风险：若单规格商品误命中，会产生冗余标题

### Slice 2

- 改动范围：兼容返回路径
- 主要文件/模块：兼容标题改写逻辑
- 预期产出：不同返回路径标题规则统一
- 风险：若只改主路径，客户端间标题会不一致

## 顺序与并行关系

- 先确定规格名前缀规则，再同步兼容路径
- 两个切片可部分并行，但建议串行，避免规则漂移
- 关键路径是“什么情况下加前缀”的统一判定

## 验证计划

- 最小验证：`go test ./entities/converters/auction_converters ./entities/loaders/product_loaders`
- 重点确认：多规格商品加前缀，单规格商品不加
- 回归确认：已有支持规格前缀的竞拍类型不回退

## 回滚与兜底

- 若标题长度或规则出现异常，先回滚兼容路径
- 再视情况回滚主路径标题改造
- 不需要额外降级开关
