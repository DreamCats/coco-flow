# 协议驱动型需求汇报模板

## 背景

- 当前有一类需求并不是普通代码需求，而是协议驱动型需求
- 这类需求通常涉及：
  - API 协议新增字段（Proto）
  - RPC 协议新增字段（Thrift）
  - 平台生成代码 / Overpass / Gateway 工作流
  - 多仓依赖升级与联调

## 为什么难

### 1. 不是单仓代码问题

- 需求可能同时涉及 `oec/live_shopapi`、`ttec/live_pack`、IDL 仓库、生成产物仓库
- 其中任一仓库没完成，整体都无法闭环

### 2. 不是纯代码动作

- 协议新增字段往往需要人工触发平台流程
- 生成代码和依赖升级存在等待时间
- LLM 无法在本地单次完成所有步骤

### 3. 存在串行阻塞链路

典型顺序：

1. 业务确认字段语义
2. 修改 Proto / Thrift 定义
3. 人工触发平台生成
4. 等待生成物
5. 升级依赖版本
6. 下游业务接线
7. 联调验证

### 4. 标准答案不只是代码 diff

- 协议变更说明
- 依赖关系说明
- 平台触发动作
- 人工交接清单
- 下游接线方案
- 联调与回滚方案

## 当前阶段建议

### 阶段 1：先识别

- 在 `refine/design/plan` 阶段识别这是不是协议驱动型需求
- 自动输出：
  - `protocol_layer`: `proto` / `thrift` / `both`
  - `blocked_by_human`
  - `blocked_by_platform`
  - `generated_artifacts`

### 阶段 2：产出可协作执行包

- 协议变更草案
- 依赖仓库清单
- 平台触发 checklist
- 下游接线 plan
- 联调 checklist

### 阶段 3：半自动接线

- 人工完成平台触发后
- LLM 继续完成依赖升级、业务接线、验证方案补齐

## 对老板可汇报的结论

- 这类需求天然是人机协作任务，不应当以“端到端自动完成”为短期目标
- 短期最有价值的能力是：
  - 能识别
  - 能拆解
  - 能明确阻塞点
  - 能产出可执行协作包
- 中期再逐步支持“生成产物回来后的半自动接线”

## 示例依赖图

```text
Client / Frontend
  -> oec/live_shopapi (Proto / API)
      -> ttec/live_pack (Thrift / RPC)
          -> downstream services
```

## 汇报用一句话

当前最难支持的不是多仓改代码，而是协议驱动型需求。它涉及 Proto、Thrift、平台生成和人工协作，天然不是单次自动化任务。短期目标不是端到端自动完成，而是先让系统能准确识别、正确拆解、明确阻塞点，并产出可协作执行包。
