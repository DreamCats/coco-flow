# 阶段产物文档边界设计

本文档定义 Refine、Design、Plan 三个阶段产出的人类可读文档的边界、内容结构和设计原则。

## 设计原则

1. **文档是给研发 review 用的，不是给机器编排用的。** 机器编排靠 JSON artifacts，文档是 JSON 的人类可读投影。
2. **每个文档有明确的 review 目标和 review 后动作。** 研发看完知道自己要做什么决定。
3. **只服务后端（Go 为主）。** 前端改动不涉及，不需要"多端协议"等 section。
4. **颗粒度从粗到细递进。** 需求确认书说"做什么"，技术方案说"怎么拆"，执行计划说"具体几件事"。
5. **待确认项是最有价值的输出。** 每个文档都应该把不确定的东西显式标注出来，驱动研发和产品/研发之间的对齐。

## 三个文档的定位

| 文档 | 给谁看 | 核心问题 | review 后的动作 |
|------|--------|---------|----------------|
| 需求确认书 (prd-refined.md) | 研发 + 产品 | 做什么、不做什么、待确认什么 | 和产品对齐待确认项，锁定需求 |
| 技术方案 (design.md) | 研发 | 怎么拆、每个仓库做什么、有没有依赖 | 确认方案合理，进入执行 |
| 执行计划 (plan.md) | 研发 | 具体几个任务、什么顺序、怎么验证 | 确认任务拆分合理，启动编码 |

---

## 一、需求确认书 (prd-refined.md)

### 当前问题

现有 5 个 section（核心诉求、改动范围、风险提示、讨论点、边界与非目标）偏产品自检清单，不是研发拿来干活的东西。缺少验收标准、场景级描述、业务流程影响等研发真正关心的信息。

### 新结构

```markdown
# 需求确认书

## 需求概述
一段话说清楚这个需求的背景和目标，面向后端研发写。
不是 PRD 的复制粘贴，而是研发视角的翻译：为什么要做这件事，做完之后业务上会有什么变化。

## 具体变更点
每个变更点用「场景 → 当前行为 → 期望行为」三段式描述。
只说业务层面要变什么，不涉及怎么实现。

示例：
- 场景：用户在竞拍中查看讲解卡
- 当前行为：讲解卡不展示竞拍态提示
- 期望行为：讲解卡在竞拍态下展示倒计时和当前出价

## 验收标准
可枚举的、可测试的条件。研发做完后对着这个清单自检。
每条用「当 X 时，应该 Y」的格式。

示例：
- 当用户处于竞拍态时，讲解卡应显示倒计时
- 当竞拍结束时，提示应自动消失
- 非竞拍态下，讲解卡行为不变

## 边界与非目标
明确不做什么。特别是容易被产品"顺手"扩大的范围。
每条说明原因。

## 待确认项
和产品需要对齐的问题。每条标注：
- 问题描述
- 当前假设（如果没有明确答案，我们先按什么假设推进）
- 影响范围（如果答案不同，会影响哪些改动）

这是研发 review 这个文档时最核心的输出——确认完这些，需求才算锁定。
```

### Section 对照

| 旧 section | 新 section | 变化说明 |
|------------|-----------|---------|
| 核心诉求 | 需求概述 | 从列表改为段落，增加背景和业务影响 |
| 改动范围 | 具体变更点 | 从模糊的"范围"改为三段式场景描述 |
| 风险提示 | (移除) | 业务风险合并到待确认项，技术风险留给 design |
| 讨论点 | 待确认项 | 增加"当前假设"和"影响范围"结构 |
| 边界与非目标 | 边界与非目标 | 保留，增加每条原因 |
| (无) | 验收标准 | 新增，从 PRD 和 intent 中提取 |

### 内容来源映射

| 新 section | 数据来源 |
|-----------|---------|
| 需求概述 | `refine-intent.json` 的 `goal` + PRD 原文背景段 |
| 具体变更点 | `refine-intent.json` 的 `change_points` + PRD 原文场景段 |
| 验收标准 | PRD 原文验收条件 + `change_points` 推导 |
| 边界与非目标 | `refine-intent.json` 的 `boundary_seed` |
| 待确认项 | `refine-intent.json` 的 `discussion_seed` + `risks_seed` 中的不确定项 |

## 二、技术方案 (design.md)

### 当前问题

12 个 section 混杂了技术方案和项目管理信息（给 QA 的输入、人力评估、实验、协议变更）。对纯后端场景冗余。研发 review 时重点不突出。

### 新结构

```markdown
# 技术方案

## 改造点总览
从需求确认书的变更点出发，列出对应的技术改造点。
每个改造点一句话说清楚做什么。

示例：
1. 竞拍态提示数据源接入：从竞拍服务获取实时竞拍状态，透传至讲解卡接口
2. 讲解卡接口扩展：response 增加 auction_hint 字段

## 总体方案
2-3 段话描述整体思路：
- 核心判断：为什么选这个方案，排除了什么替代方案
- 主链路：请求从哪进来，经过哪些服务/模块，数据怎么流转
- 关键约束：性能、兼容性、配置开关等

## 分仓库方案

### {repo_name}
- **职责**: 这个仓库在本次需求中负责什么（一句话）
- **scope_tier**: must_change / co_change / validate_only
- **主要改动**:
  - 改动 1：模块/接口/数据层面的描述，不到代码行
  - 改动 2：...
- **候选文件**: 可能涉及的文件列表（来自 repo research）
- **改动理由**: 为什么这个仓库需要改（来自 repo binding 的 reason）

### {另一个 repo_name}
...

## 仓库依赖关系
哪个仓库依赖哪个，标注依赖类型：
- 接口依赖：A 调用 B 的接口
- 数据依赖：A 读 B 写入的数据
- 配置依赖：A 读 B 管理的配置

如果有先后约束（必须先改 A 再改 B），在这里说明。

## 接口协议变更
本次需求是否涉及对外暴露的接口新增/修改（包括给前端、给其他服务）。
如果有：
- 变更接口：哪个接口、哪个字段、新增还是修改
- 变更内容：字段含义、类型、取值范围
- 下游消费方：前端 / 其他服务 / 客户端
- 是否需要和下游对齐

示例：
- 接口 GetAuctionCard response 新增 `auction_hint` 字段（string），竞拍态时返回倒计时文案，前端消费。需和前端对齐字段格式。
- 实验标记 `exp_bucket` 通过 response header 下发，前端透传至埋点，需对齐 header key 命名。

如果本次需求不涉及协议变更，写"本次需求不涉及对外接口协议变更"即可。

## 风险与待确认项
技术层面的风险和不确定项。业务风险在需求确认书里，不要重复。
每条标注影响程度（高/中/低）和建议应对方式。
```

### Section 对照

| 旧 section | 新 section | 变化说明 |
|------------|-----------|---------|
| 系统改造点 | 改造点总览 | 保留，精简为一句话列表 |
| 方案设计 / 总体方案 | 总体方案 | 保留核心，增加"为什么选这个方案" |
| 分系统改造 | 分仓库方案 | 简化结构，去掉 selection_context 等内部信息 |
| 联动验证仓库 | 分仓库方案 (scope_tier=validate_only) | 合并到分仓库方案，通过 scope_tier 区分 |
| 参考链路 | (移除) | reference_only 仓库不需要在方案中占位 |
| 系统依赖关系 | 仓库依赖关系 | 保留，增加依赖类型分类 |
| 关键链路说明 | 总体方案 (合并) | 主链路描述合并到总体方案 |
| 多端协议是否有变更 | 接口协议变更 | 改为服务端视角：关注对外暴露的接口新增/修改，包括给前端的字段、实验下发等 |
| 存储&&配置是否有变更 | 风险与待确认项 (合并) | 有变更时作为风险项提出 |
| 是否有实验，实验怎么涉及 | 接口协议变更 + 风险与待确认项 | 实验涉及协议下发的归入接口协议变更，实验配置风险归入风险项 |
| 给 QA 的输入 | (移除) | 项目管理信息，可作为可选附录 |
| 人力评估 | (移除) | 项目管理信息 |

### 内容来源映射

| 新 section | 数据来源 |
|-----------|---------|
| 改造点总览 | `design-change-points.json` |
| 总体方案 | `design-sections.json` 的 `solution_overview` + `critical_flows` |
| 分仓库方案 | `design-repo-binding.json` + `design-research.json` + `design-sections.json` |
| 仓库依赖关系 | `design-sections.json` 的 `system_dependencies` |
| 接口协议变更 | `design-research.json` 的接口分析 + `design-sections.json` 的 `protocol_changes` |
| 风险与待确认项 | `design-sections.json` 的 `risk_boundaries` + verify issues |

## 三、执行计划 (plan.md)

### 当前问题

7 个 section 更像是给调度系统看的（实施策略、并发与协同、交付边界），研发 review 时不知道该重点看什么。任务拆分的颗粒度不够具体——"goal" 经常是一句模糊的描述。

### 新结构

```markdown
# 执行计划

## 任务清单

### T1 [{repo}] {标题}
- **目标**: 一句话说清楚这个任务做什么
- **改动范围**: 涉及哪些文件/模块（从 design 的候选文件细化）
- **具体做什么**:
  - 步骤级描述，到「在 X 文件/模块中增加/修改 Y 逻辑」的颗粒度
  - 不到代码行，但要到模块+接口+字段级别
- **完成标准**: 怎么判断做完了（可执行的验证命令或可检查的条件）
- **依赖**: 依赖哪个任务先完成，为什么

### T2 [{repo}] {标题}
...

## 执行顺序
用箭头或表格展示：先做什么、后做什么、哪些可以并行。
只列有依赖关系的约束，不需要列"推荐顺序"。

## 验证策略
整体验证怎么做：
- 每个任务的独立验证（编译、单测）
- 跨仓库的联动验证（如果有依赖）
- 最终的端到端验证路径

## 风险与阻塞项
可能影响执行的风险。每条标注：
- 是否阻塞（阻塞 = 必须先解决才能开始编码）
- 建议应对方式
```

### Section 对照

| 旧 section | 新 section | 变化说明 |
|------------|-----------|---------|
| 实施策略 | (移除) | "不再重新 adjudicate repo scope" 是内部信息 |
| 任务拆分 | 任务清单 | 增加"具体做什么"步骤级描述 |
| 执行顺序 | 执行顺序 | 简化，只列约束 |
| 并发与协同 | 执行顺序 (合并) | 并行信息合并到执行顺序 |
| 验证计划 | 验证策略 | 增加跨仓库联动验证和端到端路径 |
| 阻塞项与风险 | 风险与阻塞项 | 增加是否阻塞标注 |
| 交付边界 | (移除) | 与需求确认书的"边界与非目标"重复 |

### 任务颗粒度定义

一个任务（Work Item）应该满足：
1. **单仓库**: 一个任务只涉及一个仓库
2. **可独立验证**: 做完后可以通过编译/单测独立验证，不依赖其他任务完成
3. **研发可评估**: 研发看了能判断"大概需要多久"和"方向对不对"
4. **Code agent 可执行**: 给到 code agent 后，agent 能在一个 session 内完成

**颗粒度示例**:
- 合适: "在 auction_service 的 GetAuctionCard 接口中，增加 auction_hint 字段的填充逻辑，数据从 AuctionStatusCache 获取"
- 太粗: "实现竞拍态提示功能"
- 太细: "在 auction_service/handler.go 第 142 行的 GetAuctionCard 方法中增加 if 判断"

## 四、Code 阶段输入 (code-batch)

### 当前状态

code-batch.json 传入 `work_items`（goal + change_scope + done_definition + verification_steps），code-batch.md 是其人类可读版本。agent 在 worktree 中工作，可读取 prd-refined.md / design.md / plan.md 作为背景参考。

### 设计原则

**Plan 告诉 Code "做什么 + 改哪些文件 + 怎么验证"，Code agent 自己决定"怎么改"。**

这和真实的研发流程一致：tech lead 布置任务不会写到代码行，但会明确目标和验收标准。

### 颗粒度

| 维度 | 颗粒度 | 说明 |
|------|--------|------|
| 目标 | 自然语言，到接口/字段级别 | "在 GetAuctionCard 接口增加 auction_hint 字段" |
| 改动范围 | 文件列表 | 候选文件，允许 agent 发现更多 |
| 完成标准 | 可执行的验证命令 | `go build ./...`、`go test ./pkg/auction/...` |
| 实现方式 | 不指定 | agent 读代码后自行决定 |

### 与 plan.md 任务清单的关系

plan.md 中"具体做什么"的步骤级描述会被映射到 code-batch.json 的 `work_items[].goal`。但 code-batch 是结构化的机器输入，plan.md 是给人 review 的文本——两者内容一致，格式不同。

## 五、文档间的信息流转

```
PRD 原文
  │
  ▼
需求确认书 (prd-refined.md)
  │  研发 review → 和产品对齐待确认项 → 锁定需求
  │
  │  传递给 design: 变更点、验收标准、边界
  ▼
技术方案 (design.md)
  │  研发 review → 确认方案合理 → 进入执行
  │
  │  传递给 plan: 改造点、分仓库方案、候选文件、依赖关系
  ▼
执行计划 (plan.md)
  │  研发 review → 确认任务拆分合理 → 启动编码
  │
  │  传递给 code: work_items (goal + change_scope + done_definition + verification)
  ▼
code-batch.json / code-batch.md (内部物料，不给人 review)
```

**关键约束**: 每层只传递下层需要的信息，不做全量透传。code agent 不需要知道"为什么选这个方案"，只需要知道"做什么 + 怎么验证"。
