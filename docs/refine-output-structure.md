# Refine Output Structure

本文记录 `refine` 产物面向研发阅读时的目标章节结构，以及当前引擎对这套结构的支撑情况。

如果本文与代码不一致，以代码为准。

## 目标章节

面向研发的 `prd-refined.md`，当前建议收敛为 5 个章节：

1. `变更范围`
2. `非目标`
3. `关键约束`
4. `验收标准`
5. `待确认项`

这 5 个标题对应研发最关心的 5 个问题：

1. 要改什么
2. 不要改什么
3. 哪些规则不能错
4. 改完怎么算对
5. 还有哪些不确定，先别拍脑袋

## 为什么收敛成这 5 章

这套结构的目标不是把需求写得“完整好看”，而是让研发在最短阅读路径里拿到可执行信息。

相比更偏产品表达的章节名，这 5 章有几个优势：

- `变更范围` 直接回答实现边界，研发能快速判断从哪里下手
- `非目标` 显式约束不改范围，减少误扩需求
- `关键约束` 可以统一承接业务规则、状态约束、兼容性和默认行为
- `验收标准` 把“怎么才算做完”前置，不把验证留到最后碰运气
- `待确认项` 明确告诉研发哪里不能脑补

## 当前 refine 引擎能不能支撑

结论：

- 编排方向能支撑
- 当前字段契约还没有完全对齐

也就是说，`refine` 现在的 pipeline 没有问题，但当前输出 schema 还是旧结构，离这 5 章还差一层字段和文档契约调整。

## 当前支撑较好的章节

### `变更范围`

当前 `intent` 已经会抽取：

- `goal`
- `potential_features`

再加上 knowledge brief 的术语消歧，已经具备生成“这次要改什么”的基础能力。

### `关键约束`

当前 `intent.constraints`、business memory 和 approved knowledge brief 已经能支撑：

- 业务规则
- 默认行为
- 限制条件
- 历史稳定约束

所以把现有 `constraints + 部分 business rules` 收敛成 `关键约束` 是自然的。

### `待确认项`

当前 `intent.open_questions` 已经是显式字段，`native refine` 的 verifier 也会检查缺失信息是否进入待确认问题。

这一章是当前最成熟的部分之一。

## 当前支撑较弱的章节

### `非目标`

当前没有独立的 `non_goals` 字段。

现状是：

- 一部分“不做什么”会混在 `constraints`
- 一部分会被写进旧的 `边界条件`
- 一部分完全依赖生成阶段临场发挥

所以如果直接切到 `非目标` 章节，质量会不稳定。

### `验收标准`

当前 prompt 里虽然要求输出验收相关章节，但上游没有单独的 intent scaffold 来稳定供给这类内容。

结果就是：

- native 模式下更依赖模型现场总结
- local fallback 下这一章更像模板占位

所以这是当前第二个明显短板。

## 当前实现里还存在的旧结构耦合

现在不能只改 `refine` 标题，不改下游。原因是当前还有几处硬耦合：

- `native refine` prompt 仍要求输出旧章节
- `local fallback refine` 仍写死旧章节
- `plan` 的 `parse_refined_sections(...)` 仍按旧章节名解析

因此如果后续正式切到 5 章节，需要把 `refine` 和 `plan` 一起迁移，或者至少先做新旧兼容。

## 当前推荐判断

这 5 个章节是对的，而且更适合研发消费。

但如果问“当前 refine 引擎是否已经很好支撑”，更准确的说法是：

- `变更范围 / 关键约束 / 待确认项` 已经有较强基础
- `非目标 / 验收标准` 还需要补强
- 真正落地时不能只改文档标题，必须同步调整 schema 和 `plan` 解析

## 如果继续演进，最小改动集是什么

下一步如果正式切换到这 5 章，最小改动集建议是：

1. 在 `RefineIntent` 中增加显式字段：`non_goals`、`acceptance_criteria`
2. 修改 `native refine` prompt、local fallback 和 verifier，使其以这 5 章为正式输出契约
3. 修改 `plan` 的 refined PRD 解析逻辑，消费这 5 章，或者先做新旧标题兼容映射

## 一句话结论

当前 `refine` 引擎的骨架能支撑这 5 章，但还没有完全对齐这 5 章的 schema；最弱的是 `非目标` 和 `验收标准`，真正切换时要连 `plan` 的解析一起改。
