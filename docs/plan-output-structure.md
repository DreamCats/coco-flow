# Plan Output Structure

本文记录面向研发的 `design` / `plan` 产物建议结构，重点沉淀当前讨论出来的章节设计和其中的主次关系。

如果本文与代码不一致，以代码为准。

关于执行型 `plan.md` 的建议结构，另见
[`docs/plan-task-structure.md`](docs/plan-task-structure.md)。

## 目标章节

面向研发的 `design` 文档，当前建议采用以下 7 个章节：

1. `系统改造点`
2. `方案设计`
3. `多端协议是否有变更`
4. `存储&&配置是否有变更`
5. `是否有实验，实验怎么涉及`
6. `给 QA 的输入`
7. `人力评估`

## P0 核心

这套结构里，P0 核心是前两章：

1. `系统改造点`
2. `方案设计`

原因：

- `系统改造点` 负责用少量句子建立整体认知
- `方案设计` 负责把“哪些系统做哪些事、系统之间如何依赖”说清楚

后面的协议、存储、实验、QA、人力评估，都更像专项检查项或补充决策项；重要，但不是设计骨架本身。

## 每章的职责

### `系统改造点`

这一章只做一件事：用几句话概述这次改造的核心点。

建议每个改造点都回答三件事：

- 改什么
- 为什么改
- 预期带来什么结果

这一章不要展开过多实现细节，它的作用是让研发、评审者和后续执行者先快速建立同一张全局图。

### `方案设计`

这一章是整个文档最核心的部分。

它不应该写成：

- 按仓库罗列改动清单
- 或按需求背景讲故事的大段 prose

它应该服务后续 `plan` 的任务拆分和依赖判断，因此要把下面三件事说清楚：

- 哪些系统做哪些事
- 每个系统服务于哪些核心改造点
- 系统之间的依赖关系是什么

## 为什么 `方案设计` 最难

这一章最容易写坏成两种东西：

- 写成“文件/仓库清单”，最后只有改动目录，没有设计
- 写成“需求说明重述”，最后只有背景，没有系统职责和依赖

真正难的地方不在于“写很多”，而在于把跨系统职责、上下游接口、实施顺序和关键链路讲成一张可执行的图。

## `方案设计` 的推荐内部结构

为了让这一章真正能支撑后续 `plan`，建议固定拆成 4 个子块：

1. `总体方案`
2. `分系统改造`
3. `系统依赖关系`
4. `关键链路说明`

### `总体方案`

这里回答：

- 整体方案怎么落
- 主链路从哪个系统开始，到哪个系统结束
- 为什么采用这种拆分

这是总述，不是细节列表。

### `分系统改造`

这里建议按系统或仓库逐个写，每个系统固定回答：

- 这个系统承担什么职责
- 它服务于哪个核心改造点
- 这次具体要做什么改动
- 它依赖哪些上游输入
- 它对下游输出什么

这一块写清楚后，后续 `plan` 才能自然拆成 repo / task。

### `系统依赖关系`

这里不要写代码 import 依赖，而要写“实施依赖”：

- 谁先改
- 谁后改
- 谁阻塞谁
- 哪些能并行
- 哪些是强依赖，哪些是弱依赖

这部分直接服务后续排期、分工和实施顺序判断。

### `关键链路说明`

这部分专门解释主链路：

- 请求怎么流转
- 数据怎么流转
- 状态在哪里变化
- 异常在哪一层兜底

它解决的是“系统都写了，但读者仍然串不起来”的问题。

## 其它章节为什么保留

### `多端协议是否有变更`

这一章非常重要，因为很多需求真正的风险来自上下游协议没对齐，而不是单点实现逻辑。

### `存储&&配置是否有变更`

这一章也应该独立保留。

因为这类变更经常直接影响：

- 上线方式
- 回滚策略
- 默认行为
- 兼容性

### `是否有实验，实验怎么涉及`

实验相关内容不应该埋在实现细节里，建议显式说明：

- 是否有实验
- 实验开关在哪
- 哪些链路受实验影响
- 灰度和回收策略是什么

### `给 QA 的输入`

这一章不是设计本体，但对研发协作非常有价值。

建议至少给 QA 这些信息：

- 主链路测试点
- 边界条件
- 异常路径
- 配置 / 实验组合
- 回归重点
- 最容易漏测的依赖系统

这部分更接近“研发视角的白盒测试建议”。

### `人力评估`

这一章可以保留，但更适合作为设计的附加结论，而不是技术设计核心本体。

## 一句话原则

这套结构最关键的不是章节名字本身，而是把 `方案设计` 固定成下面这个公式：

`方案设计 = 总体方案 + 分系统改造 + 系统依赖关系 + 关键链路说明`

只有这样，它才真正能支撑后续 `plan`、任务拆分和实施顺序设计。
