# Plan

## 目标

- 统一普通竞拍讲解卡成交后的状态表达
- 保证不同讲解卡返回路径下状态一致

## 前置依赖

- 无跨仓依赖
- 不涉及配置、IDL 或发布窗口变更

## 执行切片

### Slice 1

- 改动范围：主状态判定路径
- 主要文件/模块：竞拍状态判定模块
- 预期产出：成交后统一返回“成交成功”状态
- 风险：若只改主路径，其他返回路径会继续保留旧口径

### Slice 2

- 改动范围：兼容返回路径
- 主要文件/模块：旧讲解卡返回逻辑
- 预期产出：兼容路径与主路径状态一致
- 风险：若遗漏，线上会出现同一竞拍多入口状态不一致

## 顺序与并行关系

- 必须先明确主状态口径，再补兼容路径
- 两个切片不建议并行，后者依赖前者结论
- 关键路径是状态判定口径统一

## 验证计划

- 最小验证：`go test ./entities/loaders/auction_loaders ./entities/loaders/product_loaders`
- 重点确认：成交态是否统一为“成交成功”
- 同时确认：结束态仍保持原有结束口径

## 回滚与兜底

- 若兼容路径出现异常，先回滚兼容路径改动
- 本 case 可单仓完整回滚
- 不需要额外降级开关
